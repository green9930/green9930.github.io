label의 명시적 연결, 암묵적 연결

Posted on 2020-10-26 In HTML/CSS

What is Label?

label 요소란 입력 양식(form) 내부의 입력 요소(input)에 이름을 지정해주는 태그.
label과 input 사이를 연결해주는 방식은 크게 명시적 연결과 암시적 연결로 나뉜다.

  • 웹접근성 측면에서의 label
    화면리더기(screenreader)가 form 내부에서 label을 읽어 보조기술(assistive technology) 사용자가 입력해야 하는 텍스트가 무엇인지 잘 파악할 수 있게 도와준다.

  • hit area의 이점
    관련 label을 클릭해서 input을 활성화시킬 수 있으며 보다 넓어진 활성화 가능 면적은 사용자들의 편의성을 증가시킨다.
    (EX. radio나 checkbox처럼 입력 면적이 적은 곳에서 유용하게 쓸 수 있다.)

  • label 요소의 주의사항
    label 내부에 <a>, <button>와 같은 인터랙티브 컨텐츠 배치를 피한다. 이 경우 label 요소가 들어간 form 형식을 활성화하기 어렵기 때문.



label 태그의 명시적(explicit) 연결

<label> 태그의 for 속성과 <input> 태그의 id 속성을 동일하게 입력하여 명시적으로 연결시켜주는 방법
명시적 연결은 label과 input 사이에 다른 html 요소가 포함될 경우 많이 사용된다.
디자인상 편의성을 위해 명시적 연결을 권장한다.



label 태그의 암시적(implicit) 연결

<input> 태그를 <label> 태그로 감싸서 for, id 속성은 필요없이 암시적으로 연결시켜주는 방법

Casing Naming Convention 정리

Posted on 2020-10-25 Edited on 2020-10-26 In HTML/CSS

What is Case Naming Convention?

프로그래밍 중 class name을 지정할 때 사용하는 식별자 작성 방식

잘 작성된 이름은 타인뿐만 아니라 나 역시 소스 코드를 읽고 이해하는데 불필요한 수고를 덜 수 있게 해준다.
따라서 class name만 보고 용도를 파악하도록 효율적으로 정해준다면 전체적으로 생산성의 증가를 가져올 수 있다.

Case Naming Convention의 종류

  1. Camel Case
    (1) Lower Camel Case
    (2) Upper Camel Case (= Pascal Case)
  2. Pascal Case (= Upper Camel Case)
  3. Kebab Case
  4. Snake Case

  1. Camel Case
    대상의 이름을 띄어쓰기 없이 붙여서 표기하는 네이밍 컨벤션.
    단어는 모두 소문자로 작성하나 합성어의 첫 글자만 대문자로 표시하는 것이 낙타의 등(봉우리)과 비슷하다는 뜻에서 유래되었다.

    (1) Lower Camel Case (lowerCamelCase)

     첫번째 단어의 앞은 소문자로 하고 이후 합성어의 첫 글자는 대문자로 표기하는 방법

    (2) Upper Camel Case (UpperCamelCase)

     모든 단어의 첫 글자를 대문자로 표기하는 방법
 Pascal Case라고 부르기도 한다.


  2. Pascal Case (PascalCase)
    대상의 이름 중 각 단어의 첫 글자를 대문자로 표기하는 방법
    Upper Camel Case라고 부르기도 한다.



  3. Kebab Case (kebab-case)
    단어와 단어 사이에 ‘-‘ (hyphen)을 붙이는 네이밍 컨벤션으로 단어는 모두 소문자로 표기한다. 케밥을 꼬챙이에 꽂은 모양과 비슷하여 유래된 이름이다.



  4. Snake Case (snake_case)
    단어와 단어 사이에 ‘_’(underscore)을 붙이는 네이밍 컨벤션.

201018 REVIEW - How to start git

Posted on 2020-10-19 Edited on 2020-10-26 In git

[How to start git]

https://ulgoon.github.io/2019/09/09/01-git-first/

1. git init

(1) 프로젝트 폴더 만들기

  • $ mkdir sample
  • $ cd sample

(2) git init 명령으로 local repository 역할 시작

  • $ git init (= 이제부터 git을 쓰겠다)

(3) github에서 새로운 remote repository를 생성한 뒤, 복사한 주소를 local repository에 등록

  • remote repository 이름은 local repository 이름과 맞춰서 헷갈리지 않도록 한다.
  • https://github.com/example/sample.git 주소를 복사해 현재 작업공간에 remote repository 주소를 추가
    $ git remote add origin https://github.com/example/sample.git

**origin 은 복사한 주소를 부르기 위한 이름이며, 다른 이름을 써도 무방

  • 주소가 잘 추가되었는지 확인
    $ git remote get-url origin

(4) 새 파일을 만들고 작업을 수행

  • $ touch README.md
  • $ vi README.md

(5) 작업이 끝났으면 $ git status 명령으로 현재 프로젝트의 상태 확인
-> untracked files: README.md 가 보인다면 다음 작업 준비가 끝났다는 의미

(6) workspace에 추가된 새 파일을 index로 staging

  • $ git add README.md

(7) 추가된 README.md에 대한 설명을 commit
$ git commit

  • vim 에디터로 commit message 작성
  • $ git commit -m "commit message"

commit message 깔끔하게 입력하기

  • open source의 경우 모든 사용자가 읽을 수 있는 언어 선택
  • commit message의 제목에 말머리를 달아 commit의 카테고리를 바로 확인할 수 있게 함
  • feat:, docs:, test:, refactor: 등
  • Referance: Conventional commits
  • commit message의 제목은 구의 형태를 띄게 함
  • commit message는 제목과 내용이 존재
  • 내용은 optional body에 Enter로 구분하여 작성하며, 문장형으로 작성

(8) commit이 끝났으면 remote repository로 push

  • 첫 commit이면 -u(-set-upstream) flag를 붙여 push하여 현재 작업중인 branch가 업로드할 branch의 변동사항을 추적하도록 설정
    $ git push -u origin master
  • 이후에는 -u 옵션 없이 바로 push
    $ git push origin master

2. git clone

github에서 remote repository를 생성한 뒤, 작업공간으로 끌어와 작업 시작
(1) github에서 remote repository를 생성하되, LICENSE, README.md, .gitignore 등의 파일과 함께 생성하고 해당 repository 주소 복사
(2) git bash에서 $ git clone을 복사한 주소를 입력해 작업 공간에 local repository 생성
(3) 새 파일을 만들고 작업 수행
- touch README.md
- vi README.md
(4) 작업이 끝나면 $ git status 명령으로 현재 프로젝트의 상태 확인
- untracked files: README.md -> 다음 작업을 준비가 된 상태
(5) workspace에 추가된 새로운 파일을 index로 staging
- $ git add README.md
(6) 추가된 README.md에 대한 설명을 commit
- $ git commit’
(7) commit이 끝났다면 remote repository로 push
